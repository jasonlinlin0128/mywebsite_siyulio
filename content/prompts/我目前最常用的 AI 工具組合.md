對我來說，AI 工具不是越多越好，而是要能真的接進工作與筆記流程裡。

我現在比較在意的不是「哪個工具最強」，而是每個工具在我手上扮演什麼角色。只要角色清楚，工具之間就不會互相打架。

## 1. Claude Code：拿來處理實作與來回調整

如果是偏實作、要一直來回修正的工作，我現在最常開的是 Claude Code。

我會把它放在比較接近「一起做事」的位置，而不是只當查資料工具。對我來說，它最有價值的地方是：

- 幫我把任務拆成可以執行的步驟
- 幫我快速來回修正內容或結構
- 在實作過程中協助我把零散想法整理成可交付的版本

我現在最常把它用在快速 vibe coding 專案。對我來說，這種情境很適合用 agent 類工具，因為它不是只給一段答案，而是真的能陪我把專案一步一步往前推。

它比較像一個可以一起動手做事的工作夥伴。

## 2. Codex：拿來推進網站、內容與工作流實作

Codex 對我來說，很適合放在要真的把東西往前推的情境。

像我現在在做這個網站、整理內容架構、調整部署流程，就是很典型的例子。比起只問一個答案，我更常把它用在：

- 真的把某件事做完
- 把內容從草稿推到可發布
- 把技術步驟與網站結構整理清楚

所以如果 Claude Code 比較像一起來回工作，Codex 比較像把任務穩定推進的執行工具。

## 3. NotebookLM：拿來整理資料、吸收內容、建立自己的理解

NotebookLM 在我的工作流裡，比較像知識整理與吸收的工具。

它特別適合拿來面對：

- 一次有很多資料要讀
- 想先抓出重點脈絡
- 想把不同來源整理成比較容易理解的版本

對我來說，它的價值不在寫出最後版本，而在幫我先把資料消化成一個比較好思考的狀態。尤其在整理專案資料文件、吸收長篇內容、先抓出簡報脈絡時，它很有幫助。

## 4. Obsidian：拿來接住會重複使用的知識

如果只是一直跟 AI 對話，不把結果留下來，效率最後還是會流失。

所以我會把真正有複用價值的東西放進 Obsidian，例如：

- 常用 Prompt
- 工具比較筆記
- 某個工作情境的操作套路
- 寫過一次之後值得保留的模板

這樣 AI 幫我加速的是當下，Obsidian 幫我留下的是之後還能再用的東西。

## 5. 文件與試算表工具：拿來接回團隊真正會看的格式

很多內容在 AI 裡整理完，還是要回到團隊實際會看的地方。

所以我通常會把整理好的內容再放回：

- 文件工具
- 試算表
- 內部簡報
- SOP 或工作記錄

這一步很重要，因為只要沒有接回團隊日常使用的格式，AI 的輸出就很容易停留在個人層級。

## 6. 我現在比較少追求「一個工具做完全部」

以前我也會想找一個什麼都能做的工具，但現在我反而比較接受組合式工作流。

像我現在比較自然的組合會像這樣：

1. 先用 NotebookLM 吸收資料、整理脈絡
2. 再用 Claude Code 或 Codex 把東西真正往前推
3. 把有價值的內容收進 Obsidian
4. 最後整理進團隊會使用的文件格式

這樣的流程不一定最炫，但通常比較穩。

如果對照到我平常真的在做的事，大概會是：

- 用 Claude Code 快速推進 vibe coding 專案
- 用 AI 整理會議摘要、當下筆記、錄音與待辦事項
- 用 AI 改寫信件
- 用 NotebookLM 先整理專案資料文件
- 最後把簡報內容收成主管看得懂的版本

## 7. 工具選擇的標準，其實很務實

我現在挑工具，通常看這幾件事：

- 能不能幫我省下重複工作
- 產出的內容好不好再整理
- 能不能被我收進既有工作流
- 換工具之後，我的知識會不會一起被帶走

所以我會比較偏好那些可以和筆記系統、文件流程、日常工作接得上的工具，而不是只在 demo 時很厲害的工具。

## 我想建立的是一套能長期使用的 AI 工作流

我目前想做的，不是單純蒐集工具，而是慢慢建立一套自己能長期使用的組合：

- Claude Code 和 Codex 推進實作
- NotebookLM 幫我整理吸收
- 筆記系統留下方法
- 文件工具承接協作

這樣不管工具怎麼變，我真正累積下來的都不只是帳號和聊天紀錄，而是自己的工作方法。
